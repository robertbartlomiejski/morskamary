# Licensing and Compliance — Plain English Guide

This document explains the licence principles that govern all data, literature
records, and derived outputs in the morskamary repository. It is written for
researchers and contributors, not lawyers.

The authoritative governance text is `DATA_GOVERNANCE.txt`. This document
translates its key rules into plain language and explains how they apply to the
specific providers and data types in this repository.

---

## The core principle

> Treat every external document and dataset as owned by someone else,
> unless you have confirmed it is not.

This applies to:

- PDF articles from publishers (Elsevier, Clarivate, Wiley, etc.)
- Full-text content returned by APIs
- Abstracts returned by Scopus or Web of Science
- Institutional reports and EU policy documents
- Spreadsheets sourced from external organisations

When in doubt, store only the minimum — typically title, authors, year, DOI,
and URL. Do not store content you are not sure you are allowed to redistribute.

---

## What you are always allowed to store

The following metadata fields are safe to store for any literature record,
regardless of provider:

| Field | Why it is safe |
|---|---|
| Title | Bibliographic fact; not a copyright-protected text |
| Authors | Bibliographic fact |
| Year | Bibliographic fact |
| DOI | Persistent identifier; not copyrighted |
| Journal or venue name | Bibliographic fact |
| URL or DOI link | Pointer, not content |
| Subject terms / keywords | Aggregated classification, not full text |
| Source provider name | Internal metadata |
| Retrieval timestamp | Internal metadata |
| Provenance hash | Internal metadata generated by this repository |

These fields are reflected in `LiteratureRecord` and in the `allowed_metadata_fields`
list of each provider's `SourceCapability`.

---

## What you are never allowed to store

The following must not appear in any committed file, output, or derived dataset:

| What | Why |
|---|---|
| Full abstract text | Copyrighted by the publisher; redistribution requires an explicit licence from the publisher, not just an institutional access licence |
| Full article or book text | Copyrighted; never permitted without an explicit open licence (CC-BY, CC-0, etc.) |
| Raw API export payloads from Scopus, WoS, or SciVal | The provider's terms of service restrict redistribution of bulk data exports even if your institution has access |
| Author contact information (email, address) | Personal data under GDPR — prohibited by `DATA_GOVERNANCE.txt` |
| Author institutional affiliation details | May constitute personal data; institutional policies vary |
| API keys, OAuth JSON files, client secrets | Secrets — must never be committed under any circumstances |
| Researcher-level bibliometric analytics | Can identify individuals; prohibited unless fully anonymised |

If any of these appear in a committed file, open an issue immediately. See
`DATA_GOVERNANCE.txt` (section "Removing sensitive data from Git history") for
the remediation procedure.

## The three provider categories and what they mean for compliance

### Category 1 — Open providers (Crossref)

Crossref metadata is **freely redistributable**. There is no institutional
licence required, no API key required, and no restriction on storing or publishing
the returned metadata fields.

You may commit Crossref-sourced records (title, authors, year, DOI, journal, URL)
to derived datasets without further licence review.

### Category 2 — Institutional providers (Scopus, Web of Science, SciVal)

These providers require an institutional subscription. Having access through your
institution **does not mean you are licensed to redistribute the data publicly**.

The rule is:

- You may use these APIs to conduct research and verify claims during your work.
- You may store permitted bibliographic fields (see the table above) in derived
  datasets committed to this repository.
- You may not store full abstracts, full text, affiliation details, or raw bulk
  exports from these providers.
- If you are unsure whether a specific field or use is permitted, do not store it
  until you have confirmed with your institution's library or legal team.

### Category 3 — Personal and enterprise stores (Google Drive, Microsoft Graph)

These providers connect to documents you or your team own. The copyright of the
underlying documents belongs to their original creators or publishers.

Rules are the same as for open and institutional providers: only bibliographic
metadata, never full text.

Additionally: credentials for these providers are personal and must never be
committed. See `docs/RESEARCH_API_CICD_SETUP.md` for secure credential handling.

## Open-access and CC-licensed documents

If a document carries an open licence (Creative Commons CC-BY, CC-0, or similar),
you may store more than just bibliographic metadata — but only if:

1. You have confirmed the licence by reading the document header or publisher
   record, not by assuming.
2. You record the licence in the `licence_note` field of the `LiteratureRecord`.
3. The licence permits the specific use you are making (redistribution in a
   research repository).

When in doubt, treat the document as all-rights-reserved.

---

## GDPR and personal data

This repository is public. Any personal data committed here becomes publicly
accessible and cannot be easily removed.

Rules:

- Do not commit author contact information, addresses, or identifiers beyond
  name and ORCID (where ORCID is publicly declared by the author).
- Do not commit researcher-level analytics that could identify an individual's
  publication or citation history.
- Do not commit affiliation data that could be used to infer employment
  relationships without consent.

If a research step requires personal data, keep it outside the repository and
commit only anonymised or aggregated outputs. Document the anonymisation method
in a companion note file.

## What to do if you are unsure

1. **Default to storing less.** Bibliographic fields are always safe. Full text
   and analytics are almost never safe.
2. **Check the provider's terms of service**, not just the API documentation.
   API documentation describes what you can retrieve; terms of service describe
   what you can do with it.
3. **Check your institution's licence agreement** with the provider. Institutional
   licences sometimes grant additional permissions — but usually not redistribution.
4. **Ask before committing.** Open an issue or raise the question in a pull
   request comment. A delayed commit is always better than a licence violation
   that requires git history rewriting.
5. **Record the decision.** Whatever you decide, add a `licence_note` to the
   affected records and update `CHANGELOG.txt`.

---

## What happens when a rule is violated

If a committed file contains prohibited content (full abstracts, personal data,
raw proprietary exports, or secrets):

1. **Open an issue immediately** and reference this document and
   `DATA_GOVERNANCE.txt`.
2. **Do not create new commits that reference the file** while remediation is
   in progress.
3. Follow the procedure in `DATA_GOVERNANCE.txt` (section on removing sensitive
   data from Git history).
4. After remediation, update `CHANGELOG.txt` with a record of what was removed
   and why.
